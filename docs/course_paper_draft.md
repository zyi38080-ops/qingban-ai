# 陪聊机器人：详细隐私人格设定和隐私数据处理——以"晴伴AI"为例

## 摘要

随着大语言模型技术的快速发展，面向特定领域的陪伴型智能体逐渐成为人机交互的重要应用方向。本文以面向青少年游戏行为失控场景的"晴伴AI"为例，系统分析了智能体的核心组成——大语言模型、工具调用、知识库与工作流，并深入探讨了隐私人格设定与隐私数据处理的设计理念与技术实现。晴伴AI采用阿里云百炼平台Qwen3.7-plus作为核心模型，通过前端隐私脱敏、规则化风险识别、本地知识库检索、Cloudflare Worker代理等机制，构建了一套轻量化、可部署、隐私可控的陪伴与行为支持系统。系统以GitHub Pages静态网页形式部署，零后端成本，已在课程答辩中完成完整演示。本文最后结合项目实践，讨论了如何"多快好省"地使用大语言模型实现专业工作流程自动化。

**关键词**：大语言模型；智能体；隐私保护；人格设定；游戏行为支持

---

## 1. 智能体的概念与组成

### 1.1 大语言模型

大语言模型（Large Language Model, LLM）是智能体的核心引擎，负责理解用户输入、生成自然语言回复。本系统选用阿里云百炼平台的Qwen3.7-plus模型。选择该模型的原因有三：一是课程要求使用指定平台；二是Qwen3.7-plus在中文理解与生成任务上表现优异；三是通过百炼API可直接调用，无需自建推理服务。在系统架构中，LLM承担了对话生成的核心角色，但并非所有用户输入都会到达LLM——系统通过前端规则对输入进行预处理，高风险内容会被安全分流。

### 1.2 工具调用

工具是智能体超越纯文本模型的"手脚"。晴伴AI实现了五个核心工具：（1）隐私脱敏工具，基于正则表达式自动识别和隐藏手机号、邮箱、身份证号等敏感信息；（2）风险识别工具，基于关键词规则判断用户是否存在高风险表达；（3）知识检索工具，基于关键词匹配从本地知识库获取相关内容；（4）行动计划生成工具，基于模板规则为用户生成简短、具体、可执行的行动步骤；（5）本地记忆工具，在用户授权后通过LocalStorage保存偏好设置。这些工具均为前端轻量实现，不依赖后端服务。

### 1.3 知识库

知识库为智能体提供领域知识支撑，降低幻觉风险。晴伴AI采用本地JSON格式知识库，包含12条高质量知识片段，主题涵盖游戏时间管理、冲动充值、熬夜调整、亲子沟通、情绪调节、复发后恢复等场景。每条知识包含标题、关键词标签和内容正文。检索采用关键词匹配算法，每次返回最多2条最为相关的知识片段，拼接在系统提示词中作为LLM的生成参考。相比向量数据库方案，本地JSON方案虽然检索精度有限，但在12条知识的规模下几乎无性能差异，且显著降低了开发与部署成本。

### 1.4 工作流

工作流将LLM、工具和知识库串联为有序的处理管道。晴伴AI的完整工作流为：用户输入→隐私脱敏→风险识别→知识库检索→Qwen3.7-plus生成回答→行动计划生成→用户授权后保存偏好。每个环节有明确的职责边界和失败降级策略。例如，高风险输入在风险识别环节被拦截，不进入后续LLM调用；无API Key时系统自动进入Demo Mode，使用本地固定回复；知识库检索无匹配时返回空上下文。这种管道化设计使得系统行为可预测、可解释、可调试。

## 2. 晴伴AI的功能设计与技术分析

### 2.1 应用场景与专业定位

晴伴AI定位为面向青少年游戏行为失控场景的专业陪伴智能体，而非通用娱乐聊天机器人。其目标用户是那些经历"游戏时间失控、冲动充值、熬夜玩游戏、与父母因游戏发生冲突"等问题的青少年。系统在首页明确标注"本系统用于陪伴、科普与行为支持，不提供医疗诊断或心理治疗"，从定位上划清与医疗健康产品的边界。

### 2.2 系统架构

整体架构分为三层：展示层（React 19 + Vite + TypeScript，部署于GitHub Pages）、代理层（Cloudflare Worker，转发API请求并保护API Key）、模型层（阿里云百炼Qwen3.7-plus）。展示层承担所有前端工具和UI渲染；代理层为无状态函数，仅做请求转发和鉴权；模型层提供对话生成能力。三层之间通过HTTPS通信，前端不直接接触API Key。

### 2.3 隐私人格设定

人格设定是陪伴型智能体区别于通用模型的关键。晴伴AI的系统提示词明确规定：（1）不假装真人，坦诚说明自己是AI助手；（2）不冒充心理医生或咨询师；（3）不评判用户行为，不说"你肯定有网瘾"；（4）不夸大系统能力；（5）回复采用【回应】【建议】【下一步】三段式格式，确保每次回复有边界、有结构、可预期。这种设计既保护用户不产生过度依赖，也为模型行为划定了安全边界。

### 2.4 隐私数据处理

隐私保护是本系统的核心亮点。系统遵循"数据最小化"和"用户授权"两个原则。默认状态下，系统不保存任何聊天内容、真实身份信息或敏感数据。仅当用户主动在隐私设置面板中开启相关选项后，系统才通过LocalStorage保存偏好。输入脱敏在发送到LLM之前完成——前端自动识别手机号、邮箱、身份证号、银行卡号、微信号/QQ号、地址和密码关键词，并将其替换为"[已隐藏手机号]"等标记。脱敏后的文本不包含任何可追溯个人身份的信息。

### 2.5 风险识别机制

系统内置了基于关键词的风险识别模块。当用户输入中包含"自杀""自残""不想活了"等高风险关键词时，系统不调用LLM，不保存任何用户信息，直接显示预设的安全提示文本，建议用户联系可信任的家人、老师或当地紧急服务。这种设计确保了高风险场景下的响应安全性和一致性。

### 2.6 知识库与行动计划

知识库采用本地JSON格式，检索算法为关键词加权匹配（标题关键词权重2，正文关键词权重1）。行动计划工具基于5组预设计划模板，涵盖熬夜、充值、沟通、冲动控制和情绪调节五个场景。当用户输入匹配模板关键词时，系统自动展示对应的行动计划卡，用户无需手动请求。

### 2.7 Qwen3.7-plus调用方式

系统通过Cloudflare Worker作为中间代理调用阿里云百炼API。Worker存储DASHSCOPE_API_KEY环境变量，接收前端请求后转发至`https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions`，模型参数为`qwen-plus`（兼容模式调用Qwen3.7-plus），temperature设为0.7，max_tokens为800。Worker同时设置CORS头部以支持跨域请求。

### 2.8 GitHub Pages部署

前端使用Vite构建为静态文件，通过GitHub Actions自动部署至GitHub Pages。部署配置文件位于`.github/workflows/deploy.yml`，在每次推送到main分支时自动触发构建和部署。`vite.config.ts`中base路径配置为`/qingban-ai/`以匹配GitHub Pages的仓库级路径。构建时注入`VITE_DEMO_MODE=true`使部署版本默认以Demo Mode运行。

## 3. 如何多快好省地使用LLM实现专业工作流程自动化

### 3.1 使用成熟大模型降低开发成本

晴伴AI没有从零训练模型，而是直接调用Qwen3.7-plus的API。这避免了数据收集、标注、训练和部署的庞大开销。通过精心设计的系统提示词，模型行为被约束在专业陪伴的边界内，无需微调即可满足领域需求。

### 3.2 使用规则工具提升安全性

对于隐私脱敏和风险识别这两项安全关键任务，本系统选择了确定性规则而非LLM判断。正则表达式和关键词匹配是100%可预测、可审计的，不存在LLM的"漏判"或"误判"问题。对于安全功能，规则方法的可靠性优于概率方法。

### 3.3 使用本地知识库降低幻觉

LLM的知识来自训练数据，可能过时、不准确或产生幻觉。通过检索本地维护的知识库并将检索结果注入系统提示词，系统为LLM提供了"锚点"——相关信息来自经过审核的领域知识，而非模型的不稳定记忆。12条知识片段规模虽小，但每条都经过人工编写和审核，质量完全可控。

### 3.4 使用静态网页降低部署成本

GitHub Pages提供免费、零运维的静态网站托管。晴伴AI的纯前端架构天然适配静态部署——不需要服务器、不需要数据库、不需要持续运行的进程。Cloudflare Worker的免费额度（每日10万次请求）远超课程演示所需。整个系统的运营成本接近于零。

## 4. 讨论与心得

本项目从工程实践角度验证了"轻量智能体"的可行性。在12条知识、5个手工工具和1个LLM的组合下，系统能够完成专业陪伴的基本功能闭环。我们也清醒地认识到当前系统存在明确的局限：知识库规模有限、风险识别依赖简单关键词匹配（可能漏判）、行动计划模板覆盖场景有限（5类）。

对于课程学习而言，晴伴AI的核心价值不在于技术先进性，而在于它清晰地展示了智能体的四个核心组件如何协同工作，以及隐私人格设定和隐私数据处理如何从概念落实到代码。从系统提示词的三段式格式要求，到前端18条正则表达式的脱敏规则，再到"用户未授权前不保存任何记忆"的隐私原则——每一个设计决策都可以追溯到课程要求中的对应知识点。

---

## 5. 结论

本文以"晴伴AI"为案例，展示了一个面向游戏行为支持场景的轻量智能体的完整设计与实现。系统由Qwen3.7-plus大语言模型驱动，集成隐私脱敏、风险识别、知识检索、行动计划生成和本地记忆五个工具，通过本地JSON知识库提供领域知识支持，以管道化工作流串联完整处理流程。系统采用GitHub Pages静态部署、Cloudflare Worker API代理的零成本运维方案，不依赖后端服务器或数据库。隐私人格设定和隐私数据处理从代码层面体现了数据最小化和用户授权原则。该项目已完整实现，可在课程答辩中进行现场演示。
